# 嵌入式 C 语言数据结构与算法工程学习规范

## 1. 背景说明 (Overview)

在嵌入式与底层系统开发中，受限于硬件资源（RAM / ROM）与实时性要求（RTOS 调度），算法的选型直接决定了系统的稳定性与响应边界。本规范旨在定义一套“工程可用”的算法基线架构，摒弃纯竞赛向的理论算法，聚焦于可控、可预测、可维护的代码实现。

> NOTE: 嵌入式工程师学习算法的核心目的不在于应对理论测验，而在于掌握内存池管理、环形缓冲区调度以及状态机优化背后的计算思维。

## 2. 核心知识基线 (Core Knowledge Matrix)

工程师必须能够脱离任何标准模板库（如 C++ STL），仅使用原生 C 语言实现以下底层结构。

### 2.1 基础数据结构
数据结构是算法的物理载体。以下结构为第一优先级掌握内容：

- **连续内存分配**：数组、字符串
- **动态节点管理**：单向链表、双向链表
- **受限线性表**：栈 (LIFO)、队列 (FIFO)、环形队列 (Circular Buffer)
- **非线性结构**：哈希表（关注碰撞处理）、二叉树（前/中/后序遍历）

### 2.2 核心算法范式
| 类别        | 必会内容                   |
| --------- | ---------------------- |
| 查找        | 顺序查找、**二分查找（及变体）**     |
| 排序        | 冒泡、选择、插入、**快速排序、归并排序** |
| 递归        | 递归思想、递归转迭代             |
| 分治        | 二分、归并                  |
| 贪心        | 能判断“是否可贪”              |
| BFS / DFS | 图 & 树遍历                |
| 动态规划      | 一维 DP（背包是门槛）           |

### 2.3 复杂度工程评估
- **时间复杂度**：必须能准确评估 `O(N)` 与 `O(log N)` 在低频 MCU 上的执行周期差异。
- **空间复杂度**：明确“空间换时间”策略对 SRAM 造成的实际压销。

## 3. 标准化学习路径 (Learning Methodology)

> WARNING: 严禁陷入“唯数量论”的 LeetCode 刷题陷阱。死记硬背算法模板而不理解底层寻址逻辑，极易在资源受限的环境中引发堆栈溢出 (Stack Overflow)。

### 3.1 阶段一：白板实现与边界验证
使用纯 C 语言手写核心算法。
- **目标**：独立完成二分查找的 3 种边界处理、链表节点的生命周期管理。
- **验证**：能够清晰解释每一行代码的指针偏移逻辑。

### 3.2 阶段二：定向工程训练
将在线评测平台（如 LeetCode）作为验证工具，而非学习主体。
- **策略**：针对性选择 `Array`、`Binary Search`、`Two-Pointers` 标签的基础题目。
- **标准**：不追求题目数量，重点关注解答的**可复现性**与**内存安全性**。

### 3.3 阶段三：工程化模板封装
将验证通过的算法封装为高内聚的 `.c / .h` 模块，确保无未定义行为（UB）及无用的冗余代码，使其可直接合入生产代码库。

## 4. C 语言特性与算法映射指南

为将 C 语言特性与算法学习深度结合，建议参考下表的映射关系进行针对性训练：

| C 语言核心知识点 | 对应算法/数据结构 | 工程实践目标 |
| :--- | :--- | :--- |
| **循环控制** | 线性查找 | 处理空数组/单元素数组，确保 `O(N)` 复杂度。 |
| **输入输出与函数** | 二分查找 | 精确控制左右边界（左闭右闭或左闭右开），避免死循环。 |
| **一维/多维数组** | 快速排序 | 正确实现 `Partition` 分区逻辑。 |
| **指针与数组推导** | 归并排序 | 理解分治思想及辅助数组带来的空间开销。 |
| **堆内存管理 (`malloc`)**| 动态数组 (Vector) | 实现安全的扩容/缩容，严防内存泄漏。 |
| **结构体 (`struct`)** | 链表节点 | 精确管理 `next` 指针的生命周期，避免野指针。 |
| **结构体 + 指针** | 环形缓冲区 | 确保高频并发读写下指针不越界、数据不丢失。 |

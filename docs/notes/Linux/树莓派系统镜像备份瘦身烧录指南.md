# 树莓派系统镜像：完美备份、瘦身与烧录全指南

作为一名树莓派玩家，你一定遇到过这个**痛点**：
> 我用的是 128GB 的 TF 卡，即便系统只用了 2GB，备份出来的镜像文件也是雷打不动的 128GB。这不仅浪费硬盘空间，通过网络分享时更是灾难。

本教程将带你利用 Arch Linux 的原生力量，将百 G 镜像“脱水”成极小包，并教你如何完美还原。

---

## 🧠 核心原理：真空包装术

**一句话原理：** 镜像备份默认是“全盘克隆”（包含空白区域），而瘦身则是通过缩小文件系统边界并剔除无效数据块来实现。

**💡 生活中的比喻：**
想象你要寄送一个超大号行李箱（128GB TF 卡），里面只装了一件衬衫（2GB 数据）。
*   **普通备份**：直接把整个大箱子寄出去，运费昂贵且占地方。
*   **瘦身备份**：把衬衫拿出来，放进一个刚好能装下它的**真空压缩袋**里。寄送时只寄压缩袋，对方收到后再放回箱子即可。

---

## 🛠 第一阶段：全量提取 (Extraction)

在进行任何修改前，我们需要先将 TF 卡的内容原封不动地搬到电脑里。

### 1. 寻找你的 TF 卡
```bash
lsblk
```
> **为什么要执行这一步**：确认 TF 卡在系统中的设备名称（如 `/dev/sdb`），防止误删硬盘数据。

> ⚠ **警告**：请根据容量仔细辨认。如果把系统盘当成目标，你的数据将会瞬间消失！

### 2. 使用 `dd` 制作镜像
```bash
sudo dd if=/dev/sdb of=~/rpi_backup.img bs=4M status=progress
```
> **为什么要执行这一步**：`if` 是源（TF卡），`of` 是目的地（镜像文件）。`bs=4M` 能显著提升读取速度。

---

## ✂ 第二阶段：安装与瘦身 (Shrinking)

现在我们要用 `PiShrink` 给这个臃肿的文件“抽脂”。

### 1. 准备工具环境
```bash
wget https://raw.githubusercontent.com/Drewsif/PiShrink/master/pishrink.sh
```
> **为什么要执行这一步**：从 GitHub 下载核心瘦身脚本。

```bash
chmod +x pishrink.sh && sudo mv pishrink.sh /usr/local/bin/pishrink
```
> **为什么要执行这一步**：赋予执行权限并移动到系统路径，方便在任何目录下直接调用 `pishrink` 指令。

### 2. 执行“奇迹”瘦身
```bash
sudo pishrink -z ~/rpi_backup.img
```
> **为什么要执行这一步**：调用脚本处理镜像。`-z` 参数表示瘦身后顺便调用 `gzip` 进行二次压缩，让文件体积进一步大幅下降。

---

## 🔥 第三阶段：镜像烧录 (Flashing)

当你需要将瘦身后的镜像还原到新的 TF 卡，或者分发给同学时，请执行以下操作。

### 1. 准备目标卡
插上新的 TF 卡，再次确认设备路径（假设仍为 `/dev/sdb`）。

### 2. 还原镜像
```bash
# 如果是压缩过的 .img.gz 文件
zcat ~/rpi_backup.img.gz | sudo dd of=/dev/sdb bs=4M status=progress
```
> **为什么要执行这一步**：`zcat` 在不解压到硬盘的情况下直接读取压缩内容，通过管道 `|` 传给 `dd` 写入卡中。这能节省大量硬盘空间和时间。

```bash
# 如果是未压缩的 .img 文件
sudo dd if=~/rpi_backup.img of=/dev/sdb bs=4M status=progress
```
> **为什么要执行这一步**：直接将镜像文件按块写入硬件设备。

---

## ✅ 结果验证

1.  **体积对比**：输入 `ls -lh ~/rpi_backup.img*`，你会发现文件从 128G 变成了几百 MB。
2.  **启动测试**：将烧录好的卡插入树莓派。
3.  **自动扩容**：PiShrink 处理过的镜像在第一次启动时，会**自动**执行文件系统扩容，重新占满当前 TF 卡的所有空间。这意味着你用 128G 卡做的备份，可以完美还原到 16G 甚至 8G 的卡上（只要数据总量没超标）。

## 🛡 防呆设计：常见报错与避坑指南

| 现象 | 原因 | 对策 |
| :--- | :--- | :--- |
| **Permission Denied** | 没加 `sudo` | 涉及硬件底层操作和挂载，必须使用 root 权限。 |
| **pishrink 运行报错** | 缺少依赖工具 | Arch 用户请执行 `sudo pacman -S parted e2fsprogs gzip`。 |
| **dd 速度极慢** | 没设置 `bs` 或读卡器性能差 | 确保加入 `bs=4M`。如果还是慢，建议换个 USB 3.0 的读卡器。 |
| **写入后无法启动** | 镜像损坏或烧录不完整 | 写入完成后执行 `sync` 指令，确保缓存全部落盘后再拔卡。 |

> 💡 **进阶小贴士**：
> 如果你在 Arch Linux 上频繁操作镜像，建议安装 `balena-etcher` (AUR) 获得图形化界面的烧录体验，它自带校验功能，更加稳妥。
